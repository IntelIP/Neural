title: 'Data Pipeline Patterns'
description: 'Design reliable pipelines from Kalshi APIs to strategy inputs and analytics.'
---

Neural’s modular design lets you assemble data pipelines that suit different latency and resiliency requirements. This guide starts with the supported beta patterns, then calls out the broader research patterns that remain optional.

## Streaming Bot

```
Kalshi WebSocket → Strategy → OrderManager → PaperTradingClient / TradingClient
```

- Use `KalshiWebSocketClient` for low-latency quotes.
- Forward market updates directly into strategies and risk checks.
- Keep paper trading as the default onboarding path while you calibrate the execution loop.

## Research Enrichment Pipeline

```
Kalshi WebSocket / REST → MultiSourceAggregator → Research Strategy / Notebook
```

- Attach Twitter/ESPN sources only when you intentionally want the experimental research path.
- Keep multi-feed aggregation outside the default beta story until the workflow is explicitly promoted.

## Polling Dashboard

```
RESTStreamingClient → DataTransformer → Notebook / Dashboard
```

- Poll markets every second when WebSocket isn’t available.
- Normalize payloads with `DataTransformer.flatten_keys`.
- Feed the data into dashboards or quick analyses without introducing async complexity.

## Research Warehouse

```
KalshiHistoricalDataSource → Parquet/CSV → Backtester
```

- Schedule jobs (Airflow, Prefect, cron) to pull `/markets/trades` nightly.
- Persist raw responses plus enriched columns (e.g., implied probability, spread).
- Train models, run backtests, and compare strategies over consistent datasets.

## Alerting System

```
MultiSourceAggregator → Custom Handler → Slack/Webhooks
```

- Register `aggregator.data_handlers` that push notifications on specific sentiment or price thresholds.
- Use `GameSentimentTracker` only in the experimental research pipeline; it is not part of the supported default launch path.

## Reliability Tips

- Wrap collection in `asyncio.create_task` and guard with cancellation + error logs to keep services responsive.
- Use the registry to swap data sources without touching strategy code, but keep the production path centered on Kalshi-first feeds unless you have validated broader research inputs.
- Periodically run `DataBuffer.cleanup_old_data()` to prevent memory creep in long-running services.

Good data hygiene powers accurate backtests and dependable live systems. Mix and match the building blocks above to fit your latency, cost, and compliance constraints, but keep experimental ESPN/Twitter enrichment out of the default onboarding and docs path.
